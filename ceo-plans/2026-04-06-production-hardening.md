## Accepted Scope (added to this plan)

### Baseline (Hardening)

**1. Turso migration (better-sqlite3 -> @libsql/client) — PR #7 (separate)**
- **Mode: Remote Turso instance** (NOT local file). This is a re-acceptance: the prior "no Turso" decision was correct for local dev, but production on Vercel requires persistent storage outside the ephemeral filesystem.
- Provision a Turso database via `turso db create aosom-sync`
- Env vars: `TURSO_DATABASE_URL` + `TURSO_AUTH_TOKEN`
- Data migration: export current SQLite data via `.dump`, import into Turso. For fresh deploys, schema.sql runs as-is (libsql is SQLite-compatible).
- Install: `npm install @libsql/client`, `npm uninstall better-sqlite3`.
- **Known migration complexities (from outside voice review):**
  - `.exec(multiStatementSQL)` does not exist in libsql. Schema init must split into individual `client.execute()` calls per statement.
  - Variadic `.get(...args)` and `.all(...args, limit, offset)` incompatible. Must use `{ sql, args: [] }` format.
  - Result shape: libsql returns `Row` objects (array-like), not plain JS objects. All `as TypedRow` casts must be replaced with explicit row-to-object mapping.
  - **Transaction API:** `db.transaction(fn)()` in 4 places → `client.batch([...statements], "write")` for atomic writes.
  - All database functions become async (callers already use await per prior learning).
- Vitest: use local SQLite file URL (`file:./test.db`) for tests, Turso remote for production.
- **Data migration:** Export existing SQLite via `.dump`, import into Turso. Verify `shopify_product_id` linkages for 102 imported products. If lost, re-sync will mismatch.

**2. Fix Shopify rate-limit infinite recursion**
- Add `maxRetries = 3` parameter to `shopifyFetch`. When exceeded, throw a `ShopifyRateLimitError` with the last response status. Callers catch and log.

**3. Add fetch timeout on Aosom CSV download**
- Add `AbortController` with 60s timeout in `csv-fetcher.ts`. If Aosom hangs, the sync fails with a clear "CSV fetch timeout after 60s" error instead of blocking the full 300s function duration.

**4. Fix stale sync lock**
- Threshold: 30 minutes. Add `clearStaleLockIfNeeded()` in `database.ts`: query `sync_runs` for rows with `status='running'` and `started_at` > 30min ago, update to `status='failed'` with note "Stale lock cleared (timeout)".
- Call `clearStaleLockIfNeeded()` at the top of `runSync()` in `job1-sync.ts`, before `createSyncRun()`. This covers both manual trigger (API route) and cron paths since both call `runSync()`.

**5. Add try/catch on Claude JSON parse**
- In `content-generator.ts`, wrap `JSON.parse(jsonStr)` in try/catch. On parse failure: log the raw Claude response for debugging, throw `ContentGenerationError("Claude returned invalid JSON")`. Caller in import-pipeline.ts transitions job to `error` status.

**6. Remove unused @shopify/shopify-api dependency**
- `npm uninstall @shopify/shopify-api`. No transitive deps used elsewhere (verified: the raw fetch approach has no dependency on this package).

**7. Add test script to package.json**
- Add `"test": "vitest run"`. Vitest is already a devDependency.

**8. Error rescues (3 gaps from Section 2)**
- **Claude refusal:** In `content-generator.ts`, check if Claude response contains a refusal (empty content or stop_reason="end_turn" with no JSON). Transition import job to `error` with message "Claude declined to generate content for this product."
- **Facebook rate limit:** In `facebook-client.ts`, check for 429 response in `publishWithImage` and `publishText`. Return `{ success: false, error: "Facebook rate limit — try again later" }` instead of throwing.
- **Turso/DB connection failure:** In `database.ts` `getDb()`, wrap client creation in try/catch. On failure, throw `DatabaseConnectionError("Cannot connect to database")` with clear message. API routes return 503 instead of raw 500.

**9. CSP audit for production (investigation item)**
- Run `next build` with `unsafe-eval` removed from `script-src` in `next.config.ts`. If the production build works without runtime errors, keep it removed. If Next.js requires it for production middleware or polyfills, document why and keep it with a code comment. This is an investigation item, not a guaranteed fix.

### Expansions (Cherry-Picked)

**9. Stock Velocity (Fastest Sellers)**
- Add `getTrendingProducts()` function in `database.ts`:
  ```sql
  SELECT ph.sku, p.name, p.price, p.image1, p.shopify_product_id,
         SUM(ph.old_qty - ph.new_qty) as units_moved
  FROM price_history ph
  JOIN products p ON ph.sku = p.sku
  WHERE ph.change_type = 'stock_change'
    AND ph.detected_at > cast(strftime('%s','now','-14 days') as integer)
    AND ph.old_qty > ph.new_qty
  GROUP BY ph.sku
  ORDER BY units_moved DESC
  LIMIT 10
  ```
- **Caveat:** This measures Aosom supplier stock depletion, not direct sales from our store. It's a proxy for demand.
- In `src/app/api/insights/route.ts`: replace hardcoded `sellers: []` with call to `getTrendingProducts()`. Rename response key from `sellers` to `trending`.
- In `src/app/(dashboard)/dashboard-client.tsx`: update the fetch consumer to read `trending` instead of `sellers`. Rename panel title from "Fastest Selling" to "Trending Products".
- Include `shopify_product_id` in results so the existing "In store" / "Not imported" badge logic works on trending items.

**10. Compare-At Price (Sale Pricing)**
- Change `updateShopifyVariantPrice(variantId, newPrice)` signature to `updateShopifyVariantPrice(variantId, newPrice, oldPrice?)` in `shopify-client.ts`.
- When `oldPrice` is provided and `newPrice < oldPrice`: include `compare_at_price: oldPrice` in the Shopify REST payload.
- When `newPrice >= oldPrice` (or `oldPrice` not provided): include `compare_at_price: null` to clear any previous sale display.
- Call site in `job1-sync.ts` (~line 189): pass `change.oldValue` as the `oldPrice` argument. This value is already available from the diff engine.
- Applies at the variant level (Shopify REST `PUT /variants/{id}.json`).

**11. Health Check Endpoint**
- `GET /api/health` (public, no auth required for external monitors)
- Returns: `{ status: "ok"|"degraded"|"down", db: true|false, lastSync: { timestamp, status, age_minutes }, version: "<from package.json>" }`
- "degraded" if last sync failed or is >26h old (missed daily window). "down" if DB unreachable.
- No system metrics (CPU/memory). Just DB + sync status. Keep it simple.

**12. Tests (2 new test files)**
- `tests/database.test.ts`: test `getTrendingProducts()` with seeded price_history data, test `clearStaleLockIfNeeded()` with a stale sync_runs row.
- Use in-memory SQLite or temp file for test isolation.

**13. Fix outside voice findings (non-Turso)**
- `job1-sync.ts`: ensure `change.oldValue` is parsed as `Number()` before passing to `updateShopifyVariantPrice` for compareAtPrice logic.
- `stale lock`: use `datetime(started_at) < datetime('now', '-30 minutes')` for ISO text column comparison.
- `package.json` and `VERSION`: reconcile version string (use `0.1.2.0` for this release).
